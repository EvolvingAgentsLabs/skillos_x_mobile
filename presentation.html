¿Qué es un Sub-Agente?

Un agente especializado al que el orquestador delega tareas.

# .claude/agents/data-validator.md --- name: cardiac-data-validator description: Valida datos clínicos. Usar para QA de datos. model: claude-sonnet-4-6 tools: Read, Bash, Grep skills: [pointcloud-metrics] mcpServers: orthanc: type: http url: https://orthanc/mcp --- Eres un especialista en QA de datos clínicos. Dado un shape reconstruido y un point cloud, computa métricas...

Archivo Markdown

YAML frontmatter + instrucciones en texto plano. Legible y versionable.

Aislamiento de contexto

Cada sub-agente tiene su propia ventana de contexto. Solo un resumen vuelve al orquestador.

Scoping por agente

Cada uno declara sus Skills, MCP servers y herramientas permitidas.

Concepto

¿Qué es un Skill?

Instrucciones modulares en Markdown que el agente carga on-demand.

¿Por qué "Skill"?

Como las habilidades humanas: no piensas en todas a la vez. Cuando necesitas cocinar, activas tus habilidades de cocina. Cuando necesitas conducir, activas las de conducir.

El agente funciona igual: ve un catálogo de habilidades disponibles y decide cuál cargar según la tarea.

El mecanismo

Paso 1: System Prompt
Solo ve nombre + descripción corta
medication-reminder: "Verifica horario..."
↓ El agente decide que lo necesita
Paso 2: Tool Call
load_skill("medication-reminder")
↓ Se inyectan las instrucciones completas
Paso 3: Contexto expandido
35 líneas de protocolo detallado
Progressive disclosure — solo lo que necesita
Concepto

¿Qué es Memory?

Almacenamiento persistente en archivos Markdown que sobrevive entre sesiones.

¿Por qué "Memory"?

Porque funciona como la memoria episódica humana: el agente recuerda lo que vivió, lo que observó, y lo que otros le dijeron.

No es un database — es conocimiento contextual que el agente lee, escribe y actualiza con sus propias palabras.

El mecanismo

1. Stores — Carpetas temáticas con manifiesto

health-profile/, daily-log/, consolidated/

2. Documentos Markdown — Legibles por humanos

El agente escribe en lenguaje natural

3. Versionado SHA256 — Integridad garantizada

Backup automático: v1 → v2 → v3...

4. Auto-inyección — Manifiestos en el system prompt

El agente sabe qué memorias existen sin buscar

Concepto

¿Qué son los Dreams?

Consolidación offline donde un LLM reorganiza las memorias del agente.

¿Por qué "Dreams"?

Porque replica lo que hace el sueño biológico:

  • SWS — Replay de memorias recientes
  • REM — Reorganización, nuevas conexiones
  • Consolidación — Corto plazo → largo plazo

El paralelo no es metafórico — es funcional. Mismo proceso, mismo resultado.

El mecanismo

Input
Memorias existentes + transcriptos de sesión
Todo lo que el agente vivió y escribió
↓ LLM pass offline
Proceso
El LLM analiza, deduplica, categoriza
Conecta información, extrae insights
↓ Estructura emergente
Output
Documentos organizados por categoría
La estructura EMERGE del LLM — no la definimos nosotros

Corre entre sesiones — no durante la operación

Dreams = el loop humano de mejora, ahora autónomo

Antes — Loop humano

1. Ingeniero lee los traces / postmortems
↓ atención humana
2. Detecta patrones vs casos aislados
↓ juicio de síntesis
3. Refactoriza runbook / prompts / config
↓ deploy manual
Ese paso de síntesis era una caja negra — en la cabeza del ingeniero

Ahora — Dream Engine

1. Lee memory store + 1-100 session transcripts
↓ atención automática
2. LLM deduplica, fusiona, extrae insights
Dirigible con instructions
↓ output store separado
3. Produce nuevo store reorganizado
Input nunca se modifica — es un PR, no un auto-merge
↓ gate humano para adoptar
Y el propio dream genera su trace auditable via session_id

El cierre del círculo: observabilidad recursiva

La diferencia conceptual fina respecto al humano.

Dreams es batch, no real-time

El loop humano tenía dos modos: real-time (alerting, auto-rollback) y batch (la retro semanal, el refactor del wiki).

Dreams es explícitamente batch/offline: corre asíncrono, tarda minutos a decenas de minutos. Es consolidación tipo replay hipocampal durante el sueño.

Dos trabajos tácitos externalizados

  • Atención — alguien tenía que leer los logs. Ahora se procesan 1 a 100 sesiones por corrida.
  • Juicio de síntesis — ¿patrón real o caso aislado? Dirigible con el campo instructions.

Observabilidad sobre la observabilidad

¿Por qué "Dreams"? — La analogía biológica

🧠 Sueño biológico

  • Slow-Wave Sleep: Replay de memorias recientes
  • REM: Reorganización, nuevas conexiones
  • Consolidación: Corto plazo → largo plazo
  • Resultado: Mejor recall, patrones emergentes

🤖 Dream Engine

  • Input: Transcripts recientes (replay)
  • LLM pass: Reorganiza, conecta (REM)
  • Output: Docs consolidados (largo plazo)
  • Resultado: Conocimiento estructurado
El paralelo no es metafórico — es funcional. Mismo proceso, mismo resultado.

Las dos grandes preguntas abiertas

Temas para la conversación — no tenemos todas las respuestas.

Seguridad

  • Memory poisoning — ¿qué pasa si un agente escribe información falsa en la memoria compartida?
  • Skill injection — ¿cómo validar que un skill de terceros no es malicioso?
  • Dream manipulation — ¿se puede manipular la consolidación para cambiar el comportamiento del agente?
  • Autonomía vs control — ¿cuánta autonomía le das a un agente que controla otros agentes?

Escalabilidad

  • Memory at scale — ¿qué pasa cuando un agente tiene 10.000 documentos consolidados?
  • Dream compute — consolidar 1000 sesiones de 50 agentes — ¿cuánto LLM compute necesitás?
  • Skill discovery — con 10.000 skills disponibles, ¿cómo elige el agente sin explotar el contexto?
  • Coordination overhead — ¿cómo escala la comunicación inter-agente cuando hay cientos?
Estos son los problemas que definen la próxima generación de infraestructura de agentes.
